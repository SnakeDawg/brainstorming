# report

The narrative layer on top of the evidence layer. Everything `report`
writes is a pure function of files already on disk under
`memory/runs/<run-id>/` and the target's `tests.md` / `rubric.md`. You
can re-run `report` at any time to regenerate a stale report without
re-invoking any LLM or re-running the loop.

## Invocation

```
report --run <run-id>                 # writes REPORT.md only
report --run <run-id> --history       # also appends to HISTORY.md
report --run <run-id> --index         # also appends to memory/INDEX.md
report --run <run-id> --all           # all three (default at end of a loop)
```

End-of-loop default: `--all`.

## The five questions a report must answer

`REPORT.md` is designed around these — if a section doesn't serve one
of them, it doesn't belong:

1. **Did it work?** → verdict banner + TL;DR
2. **By how much?** → snapshot table + score progression chart
3. **What changed?** → diff summary + before/after samples
4. **Can I trust it?** → regression count, samples per test, evidence links
5. **What should I do next?** → follow-ups, rubric gaps, advisory flags

## Output: `memory/runs/<run-id>/REPORT.md`

Section order (fixed):

```
# Improver Run Report — <target>

**Verdict: <ACCEPT|REJECT>** · baseline <X> → **<Y>** (<±Δ>) · <N> regressions · <duration> · $<cost_estimate_usd>

## TL;DR
<1–3 sentences pulled from result.md and the accepted verdict>

## Snapshot
<table: target, objective, started/ended, iterations, accepted candidate,
 samples per test, total invocations, rubric version,
 cost: tokens_in / tokens_out / time / est_usd>

## Score progression
<ASCII bar chart, one line per iteration, using Unicode block glyphs
 (█▌░) computed from candidate_score>

## Per-test results
<table: id, name, baseline, final, samples, status>

## What changed
<link to accepted diff, 1-paragraph plain-English summary>
<one-sentence summary of proposer intent (from reasoning_chain in diff.md)>

## Before / after samples
<table: one row per test, columns: input, before, after>

## Lessons learned
<bulleted, pulled from lessons.md; top 3 if many>
Format: **iter N (cand NNN, verdict):** Intended: <goal>. Actual: <outcome>. Lesson: <insight>.

## Next steps
<follow-up run ideas, rubric gaps noticed, advisory critic flags>

## Evidence
<links to every raw file under the run directory>
```

### Bar chart format

```
baseline   0.40  ████░░░░░░
cand 001   1.00  ██████████  ✓ accept    (+0.60)
cand 002   0.75  ███████▌░░  ✗ reject    (regression on t1)
cand 003   1.00  ██████████  — no delta
```

Ten columns of width, half-blocks for finer resolution:

```
full  = floor(score × 10)
half  = (score × 10) − full ≥ 0.5
bar   = "█" × full + ("▌" if half else "") + "░" × (10 − full − (1 if half else 0))
```

## Output: `agents/<target>/HISTORY.md` (append row)

First invocation also writes the header and banner.

```markdown
> Auto-generated by improver. Do not edit by hand — changes will be overwritten.

# Improvement history for <target>

**Current score: <X>** · first run <date> · <N> runs total · <M> accepted · net **<±Δ>**

## Score curve

<plain-text ASCII curve, one line per run>

## Recent runs

<table, most recent first, linking to each REPORT.md>
```

Only accepted runs change "Current score". Every run (accept or reject)
gets a row in "Recent runs" and a point on the curve.

If a run is the first after a rebaseline, the row is annotated:
`[rebaseline vN → vM]` so the curve shows where rubric rules changed.

## Output: `memory/INDEX.md` (append row)

Repo-wide one-line-per-run table, most recent at the top.

```markdown
> Auto-generated by improver. Do not edit by hand — changes will be overwritten.

# Improvement index

| date       | target         | objective          | delta | verdict | cost     | report |
|---|---|---|---|---|---|---|
| <date> | <target> | <objective> | <±Δ> | accept/reject | $<est> | [↗](runs/<id>/REPORT.md) |
```

## Live console output (during a run)

`report` also defines the live status format that `run` / `score` /
`reflect` print between iterations:

```
▶ improver run agents/greeter
  target    · agents/greeter
  objective · handle empty input
  tests     · 3 (from agents/greeter/tests.md)
  samples   · 3 per test

  baseline  · 0.40  ████░░░░░░

  iter 1/10
    propose · drafting candidate 001 ... done
    run     · 9 invocations (3 tests × 3 samples) ...
    score   · t1 1.00  t2 1.00  t3 1.00  →  1.00  ██████████
            · delta +0.60, regressions 0
            · verdict ACCEPT
    commit  · applying 001.diff.md to format-greeting/SKILL.md
            · baseline_score: 0.40 → 1.00 in rubric.md

  ─────────────────────────────────────────────────
  DONE  0.40 → 1.00  (+0.60)  0 regressions  4m 22s
  report  · memory/runs/<id>/REPORT.md
  history · agents/greeter/HISTORY.md
  index   · memory/INDEX.md
```

This is specification, not implementation — the loop emits text
matching this shape. Formatting is plain ASCII/Unicode so it works in
any terminal.

## Determinism and regeneration

- `report` performs zero LLM calls. It is a string-assembly function
  over files.
- Editing the template in this SKILL.md and re-running `report` on a
  historical run rewrites that run's `REPORT.md` in place.
- Backfilling old runs after a template change is a tight loop:
  `for d in memory/runs/*; do report --run "$d"; done`.

## Hard rules

1. `report` never writes to `tests.md`, `rubric.md`, or any target
   source file.
2. `HISTORY.md` and `INDEX.md` carry a "do not edit" banner at the top.
   They are append-only from the loop's perspective.
3. If `REPORT.md` already exists for a run, it is **overwritten**, not
   appended — the report is always the latest render of the evidence.
