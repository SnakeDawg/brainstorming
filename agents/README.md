# Agents

Each subdirectory here is one agent. An agent is defined by an `AGENT.md`
file with YAML frontmatter that declares which skills it uses and — since
MVP-0 — how it participates in the self-improvement loop.

> **Note:** This is a demo layout. It is **not** wired into Claude Code's
> native `.claude/skills/` auto-discovery — it's a plain-text scaffold a
> human or a future meta-agent can read and edit.

## Layout

```
agents/
  <agent-name>/
    AGENT.md                 # agent definition
    rubric.md                # evaluation rules + weights + acceptance criterion
    HISTORY.md               # append-only improvement trail (auto-generated)
    skills/                  # (optional) dedicated skills owned by this agent
      <skill-name>/
        SKILL.md
```

## `AGENT.md` frontmatter schema

```yaml
---
name: <kebab-case>
description: <one sentence describing the agent>
dedicated_skills:
  - <relative path to a SKILL.md directory>
shared_skills:
  - <relative path to a SKILL.md directory under ../../skills/shared/>
improvement_agent: ../../agents/improver      # required
rubric: ./rubric.md                           # required (scoring contract)
---
```

- `dedicated_skills` are co-located under this agent's own `skills/`
  folder and should not be referenced by other agents.
- `shared_skills` point at entries under
  [`../skills/shared/`](../skills/shared/) and may be pulled in by any
  agent.
- `improvement_agent` points at [`./improver`](./improver/AGENT.md).
  Every agent in this scaffold participates in the self-improvement
  contract by declaring this field.
- `rubric` points at the scoring contract — the file that contains every
  `~~~test~~~` evaluation rule, each with its own `weight`, plus the
  acceptance criterion and any advisory critics.

## The self-improvement contract (hard rule)

**Every agent MUST ship `rubric.md`.** The `improver` refuses to run the
improvement loop against an agent whose `rubric.md` has no valid
`~~~test~~~` rule blocks or whose weights don't sum to 1.0.

Bootstrap is the standard entry point for new agents:

```
improver bootstrap agents/<new-agent>
```

Bootstrap writes a `rubric.md` *template* (with empty `~~~test~~~` rule
stubs) if none exists and waits for the human to fill in the rules. **It
never drafts rules with an LLM.** See
[`improver/skills/bootstrap/SKILL.md`](./improver/skills/bootstrap/SKILL.md)
for the full gate flow.

### Exemption

`improver` itself ships a deferred `rubric.md` because its behavior is
tested indirectly by the outcomes of running it against other agents.
The `self_improvable: false` flag in `improver/AGENT.md` gates
self-targeting off in MVP-0.

## Adding a new agent

1. Create `agents/<new-agent>/AGENT.md` with all required frontmatter
   fields above.
2. Run `improver bootstrap agents/<new-agent>`. You will be prompted to
   fill in `rubric.md` evaluation rules.
3. Fill in at least 3 `~~~test~~~` blocks using the deterministic match
   types documented in
   [`../skills/shared/execute/SKILL.md`](../skills/shared/execute/SKILL.md).
   Weights on the rules must sum to 1.0.
4. Re-run bootstrap. It validates the rules and measures the first
   baseline, writing `baseline_score` into `rubric.md` frontmatter.
5. If the agent has a dedicated skill, add it under
   `agents/<new-agent>/skills/<skill-name>/SKILL.md` (dedicated skills
   inherit their owning agent's rubric).
6. Add a row to the "Current agents" table above.

## Do not hand-edit

- `HISTORY.md` — auto-generated by `improver/skills/report` on every run.
- `rubric.md` *frontmatter field* `baseline_score` — updated by the
  loop on accept. The rest of `rubric.md` (rule blocks, weights,
  critics, acceptance criterion) is human-editable.
